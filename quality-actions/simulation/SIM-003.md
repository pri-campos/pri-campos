# SIM-003 — Idempotency with Payload Drift

## Objective

Validate whether the system correctly enforces idempotency when repeated requests carry different payloads, ensuring consistency and rejection of conflicting operations.

## How to simulate

- Resend the same idempotency key with modified payload
- Change values such as amount, currency, or metadata
- Repeat requests across different components

## What it stresses

- idempotency enforcement
- request validation logic
- authority over transaction state

## Expected invariants

- Same idempotency key must not produce different outcomes
- Conflicting payloads must be rejected
- System must preserve the original source of truth

## Failure signals

- same key producing different results
- silent acceptance of conflicting payload
- inconsistent transaction state

## Related risks

- Architecture / lack of single source of truth
- Integration / inconsistent request handling
- Business-financial / inconsistent monetary effect

## Notes

This pattern is essential for preventing financial inconsistencies in retry and duplication scenarios.
