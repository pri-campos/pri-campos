# SIM-006 — Retry Reprocessing Across Steps

## Objective

Validate system behavior when retries occur at different stages of a transaction flow.

## How to simulate

- Trigger retries at multiple points in the flow
- Replay failed transactions
- Combine retries with concurrency

## What it stresses

- retry logic
- idempotency across stages
- state transition consistency

## Expected invariants

- Retries must not create duplicate effects
- System must resume safely from failure points
- State transitions must remain valid

## Failure signals

- duplicated processing
- inconsistent state transitions
- retry loops or deadlocks

## Related risks

- Architecture / lack of idempotency enforcement across components
- Architecture / ambiguity of state ownership during retries
- Architecture / inconsistent coordination of state transitions
- Architecture / multiple writers causing duplicated effects
- Reliability / retry misbehavior
- Data consistency / duplicated state
- Architecture / lack of coordination

## Notes

Retries are common in distributed systems and must be handled safely.
